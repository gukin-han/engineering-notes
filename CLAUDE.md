# engineering-notes — 작성 규칙

이 레포 = **내 지식 바운더리 지도.** 백엔드 지식을 확장하면서, 내가 어디까지 알고 어디부터 모르는지 경계를 눈에 보이게 둔다. 망각으로 "뭘 물어볼지"를 잃지 않는 게 목적.

## 노트 포맷

- 제목(`#`)과 헤더(`##`)는 **주제형 명사구** (예: `IoC 컨테이너가 필요한 이유`, `빈 생성 시점`). 질문형 아님.
- 한 주제를 **개념 → 문제 상황 → 코드 예제 → 정리** 흐름으로 풀어간다. 이해에 필요한 만큼 설명하되 군더더기·중복은 뺀다.
- **코드 예제로 보여준다.** 특히 문제 코드와 해결 코드를 대비시키면 이해가 빠르다.
- 나열할 땐 불릿, 설명할 땐 짧은 문장. 한 문장에 기호(`·` `→` `=`)를 몰아넣지 않는다.
- 섹션은 `---`로 나누고 블록 사이는 빈 줄로 띄운다. 빽빽하면 안 됨 — 읽기 편한 게 1순위. 읽다 힘들면 그 노트가 틀린 것.
- 섹션이 4개 이상이면 제목 바로 아래에 `## 목차`로 각 헤더 링크를 단다.
- 끝은 `## 정리`로 핵심을 한 번 더 압축한다.
- 목적은 **이해와 실무 적용**이지 면접 암기가 아니다. "왜 그런가, 언제 쓰나, 뭘 조심하나"에 닿게.
- 한 노트 = 한 주제. 커지면 쪼갠다.

## 구조

- 위치: `{도메인}/{NN}-{주제}.md`. 도메인 폴더를 **루트에 바로** 두고, 파일엔 학습 순서대로 번호를 붙인다 (예: `spring-core/01-ioc-container.md`).
- 도메인 폴더는 소문자, `README`/`CLAUDE`는 대문자 → 정렬상 항상 끝에 박힘.
- 이미지·외부 raw 자료는 *필요할 때* 도메인 폴더 안에 만든다 (예: `spring-core/images/`). 안 쓰면 안 만든다.

## README (허브)

- README는 **로드맵/허브만**. 작성 규칙(이 파일)은 안 섞는다.
- 새 노트 추가·완성 시 README 로드맵에 링크 + 상태 갱신 (⬜→✅).
- `⬜` 항목 = 아직 안 잡은 지식의 가장자리. 함부로 지우지 말 것 — 그게 바운더리다.

## Claude 작업 방식

- 기본: **Claude가 핵심만 뽑아 초안 → 유저가 큐레이션** (빼기 / 더 파기 / 경계 조정). "지식은 AI에, 무엇을 담을지·어디가 한계인지는 유저가."
- 한 번에 쏟지 않는다. 유저가 부르는 속도대로 **한 주제씩**.
- 설명은 이해에 필요한 만큼만. 군더더기·중복은 뺀다.
